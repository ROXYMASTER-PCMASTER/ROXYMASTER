1. RESUMEN EJECUTIVO
================================================================================
El servidor implementa una arquitectura de WebSocket para gestionar múltiples
PCBOTs (clientes) con perfiles conectados, automatizando la apertura de URLs
y generación de comentarios IA mediante Ollama (Llama 3.2).

Estado: FUNCIONAL pero con riesgos críticos en manejo de conexiones.


2. ARQUITECTURA GENERAL
================================================================================
├─ Servidor WebSocket (asyncio + websockets)
├─ Sistema JARVIS: IA generadora de comentarios
├─ Gestor de perfiles: Mapeo de perfiles por PCBOT
├─ Grupos de tareas: Múltiples URLs con perfiles asignados
└─ Consola Admin: Interfaz CLI para comandos

[CRÍTICO] 3.1 - Manejo deficiente de excepciones en manejar_conexion()
────────────────────────────────────────────────────────────────────────
LÍNEA: 146-169
PROBLEMA:
  - Excepción genérica "except: pass" oculta todos los errores
  - No se registran desconexiones inesperadas
  - Sin diferenciación entre desconexión normal y error
  - Puede causar pérdida silenciosa de perfiles

RIESGO:

  - Imposible diagnosticar problemas de conexión
  - Debería generarse alerta si desconexión es anómala

RECOMENDACIÓN:
  try:
      msg = await asyncio.wait_for(ws.recv(), timeout=30)
      data = json.loads(msg)
      if data.get("type") == "identify":
          # ... código
  except asyncio.TimeoutError:
      logger.warning(f"Timeout en handshake: {cid}")
  except json.JSONDecodeError:
      logger.error(f"Mensaje inválido recibido")
  except websockets.exceptions.ConnectionClosed:
      logger.info(f"Conexión cerrada normalmente: {cid}")
  except Exception as e:
      logger.error(f"Error inesperado: {e}", exc_info=True)


[CRÍTICO] 3.2 - Race condition en límpio de perfiles
────────────────────────────────────────────────────
LÍNEA: 165-169
PROBLEMA:
  - No hay sincronización entre búsqueda y eliminación de perfiles
  - Si tarea_enviar_comentarios() intenta acceder mientras se limpian,
    puede fallar con KeyError
  - `perfiles_map` se modifica sin lock/semáforo

ESCENARIO:
  1. manejar_conexion() comienza a limpiar perfiles del dict
  2. Simultáneamente, tarea_enviar_comentarios() itera sobre perfiles
  3. KeyError: perfil_key eliminado durante iteración

RECOMENDACIÓN:
  Implementar threading.Lock():
  
  lock_perfiles = threading.Lock()
  
  # En manejar_conexion:
  with lock_perfiles:
      keys = [k for k, v in perfiles_map.items() if v["pcbot"] == cid]
      for k in keys:
          del perfiles_map[k]
  
  # En tarea_enviar_comentarios:
  with lock_perfiles:
      for perfil_key in grupo["perfiles"]:
          if perfil_key in perfiles_map:
              # ... enviar


[CRÍTICO] 3.3 - Falta de heartbeat/keep-alive
────────────────────────────────────────────────
LÍNEA: 161-163
PROBLEMA:
  - "async for _ in ws:" espera pasivamente mensajes
  - No hay mecanismo de ping/pong para detectar conexiones muertas
  - Puede haber conexiones TCP establecidas pero inactivas
  - Servidor no sabe si PCBOT sigue vivo

RIESGO:
  - Perfiles quedan asignados a PCBOTs no responsivos
  - Imposible hacer reintentos automáticos
  - Consume socket sin uso productivo

RECOMENDACIÓN:
  async def manejar_conexion(ws):
      cid = None
      try:
          # ... handshake
          # Crear task de heartbeat
          heartbeat_task = asyncio.create_task(heartbeat(ws, cid))
          try:
              async for msg in ws:
                  # procesar mensaje
          finally:
              heartbeat_task.cancel()
      finally:
          # ... limpieza
  
  async def heartbeat(ws, cid, interval=30):
      try:
          while True:
              await asyncio.sleep(interval)
              await ws.ping()
      except:
          logger.warning(f"Heartbeat falló para {cid}")

  async def limpiar_grupos_expirados():

                      urls_expiradas.append(url)
              
              for url in urls_expiradas:
                  await ejecutar_detener(url)

[ALTO] 3.5 - Manejo inseguro de JSON en enviar()
─────────────────────────────────────────────────
LÍNEA: 127-133
PROBLEMA:
  - "except: pass" silencia errores de serialización
  - Si `data` no es JSON-serializable, falla silenciosamente
  - No diferencia entre error de datos y error de red

RECOMENDACIÓN:
  async def enviar(pcbot_id, cmd, data):
      if pcbot_id not in pcbots:
          logger.warning(f"PCBOT {pcbot_id} no conectado")
          return False
      
      try:
          msg = json.dumps({"type": cmd, "data": data})
          await pcbots[pcbot_id].send(msg)
          return True
      except json.JSONEncodeError as e:
          logger.error(f"Error JSON: {cmd} - {e}")
          return False
      except websockets.exceptions.ConnectionClosed:
          logger.info(f"PCBOT {pcbot_id} desconectado")
          # Limpiar referencia
          pcbots.pop(pcbot_id, None)
          return False
      except Exception as e:
          logger.error(f"Error envío: {e}")
          return False
